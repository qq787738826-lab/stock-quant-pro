# Stock Quant Pro Codex 开发规则

## 项目状态

- 当前稳定版本：1.3.1
- 目标版本：1.4.0
- 当前集成分支：feature/1.4.0-agent-team
- 当前开发目标：股票智能体团队
- 智能体团队的当前阶段、真实能力与下一阶段入口，以 `docs/agent-team/CURRENT_STATE.md` 为唯一事实来源。

## 安全限制

- 不读取、显示、修改或提交 .env
- 不输出数据库真实密码或 API Key
- 不提交 node_modules、target、.venv、history-cache
- 不连接真实券商自动下单
- 不控制证券客户端
- 不自动使用真实资金
- 所有交易操作必须人工确认
- AI 不得直接修改账户现金、持仓和成交记录

## Git 限制

- 不直接修改 master
- 集成分支为 feature/1.4.0-agent-team；开发在用户指定、从集成基线创建的任务分支进行
- 开发 Codex 在一个已授权大阶段的实现、测试和检查全部通过后，应提交并普通 push 当前任务分支
- 开发 Codex 不得自行 merge；merge 必须由用户最终批准
- 禁止 git reset --hard
- 禁止 git clean -fd
- 禁止 git push --force
- 禁止删除 Git 历史

## 阶段开发治理

- 总路线图只控制方向、依赖和验收门槛，不授权自动实施全部阶段。
- 每次执行一个较大的完整开发阶段。一个大阶段可以包含多个内部工作包；内部工作包不要求分别开发、Review、提交或验收，开发 Codex 应在同一任务分支连续完成。
- 阶段限制业务范围，不限制完成授权范围所需的文件数量。普通编译、类型、测试、局部重构、常规实现选择和文档一致性问题由开发 Codex 自主修复。
- 规划、任务书或路线图中的能力不得描述成已实现；真实状态仍以 `docs/agent-team/CURRENT_STATE.md` 为准。
- `.ai/` 是外部未跟踪目录，不读取、不修改、不删除、不暂存、不提交。

### 双窗口职责

#### ChatGPT 窗口

- 规划较大的完整开发阶段，定义系统架构、阶段范围、业务边界和验收标准。
- Codex 提交并推送任务分支后，必须基于实际 Git commit、完整 diff 和测试证据进行验收，向用户明确汇报通过或不通过。
- 验收不通过时，给出针对当前大阶段的修复要求；验收通过后才规划下一大阶段。
- 不逐文件遥控 Codex，不把未提交工作区或聊天自述当作最终提交验收依据，也不代替用户批准 merge。

#### Codex 开发会话

- 一次完整执行一个已经授权的大阶段，并在同一任务分支连续完成全部内部工作包。
- 在授权范围内自主修改代码、测试和阶段文档，自主编译、测试并修复普通问题；不因内部工作包完成、文件数量或常规实现选择而暂停。
- 全部实现、测试、文档和 Git 检查通过后，精确暂存本阶段文件，commit 并普通 push 当前任务分支，供 ChatGPT 基于实际提交验收。
- 不自行 merge，不根据总路线图或已完成工作包自行进入下一大阶段。
- 遇到下述高风险暂停条件时停止相关实施并报告。

#### 用户批准边界

- 用户对必须人工选择的高风险业务规则和重大方案分歧作出最终选择。
- 用户最终批准 merge。
- 用户不需要为大阶段中的普通文件修改、内部工作包完成、commit 或普通 push 逐项批准。
- 当前大阶段合并前，开发 Codex 不得自行开始下一大阶段。

### 全局暂停门

普通编译错误、测试失败、局部重构、文件数量和常规实现选择由开发 Codex 在已授权范围内自行解决。仅在出现以下任一事项时暂停：

- 核心架构存在重大方案分歧；
- 数据库核心模型变化或不可逆迁移；
- Java/Python 公共契约的重大选择；
- 无法证明可消除的前视偏差；
- 真实交易、真实下单或真实资金操作；
- 必须由用户选择的高风险业务规则。

暂停报告必须说明触发原因、影响范围、可选方案、兼容性和迁移风险、是否需要用户选择，以及是否需要调整大阶段架构、范围或验收标准。未获得所需选择或新的明确授权前，不得继续相关高风险实施。

标准阶段流程固定为：

1. ChatGPT 基于仓库事实规划一个较大的完整开发阶段，冻结架构、范围、边界和验收标准；
2. Codex 开发会话在一个任务分支连续完成该大阶段的全部内部工作包；
3. Codex 自主编译、测试、修复普通问题并同步阶段文档；
4. Codex 完成最终 diff、测试和范围检查后 commit 并普通 push 当前任务分支；
5. ChatGPT 基于实际 Git commit、完整差异和测试证据验收，向用户报告通过或不通过；
6. 验收不通过时，Codex 在同一任务分支修复、重新测试、commit 并 push，再由 ChatGPT 验收；
7. 验收通过后，由用户最终批准 merge；
8. 合并后停止，不由 Codex 自行开始下一大阶段；ChatGPT 可以规划下一大阶段。

## 数据库限制

- 不修改已执行的 V1-V8 Flyway 文件
- 经大阶段架构明确批准的新结构必须创建新的后续 Flyway 迁移
- 数据库语法必须兼容 PostgreSQL 16
- 禁止删除或清空现有业务数据

## 股票业务限制

- 行情只能来自本地数据库或已配置数据源
- 不得编造价格、指标、公告或新闻
- 智能体结论必须引用真实证据
- 风控智能体拥有否决权
- AI 不得承诺收益
- AI 结果只用于研究和模拟交易
- 当前市场范围为沪深主板
- 排除创业板、科创板、北交所、ST 和退市股票

## 开发要求

- 每次完整完成一个已授权的大阶段；内部工作包连续执行，不建立不必要的小阶段停顿
- 修改前先阅读现有代码
- 优先复用现有实现
- 不做与任务无关的大范围重构
- 不随意增加生产依赖
- 修改后必须列出修改文件
- 修改后必须运行相关测试
- 测试失败必须明确说明
- 不允许用固定模拟数据掩盖错误

## 测试要求

Java 修改后：
- 读取项目实际 Maven 配置
- 运行相关模块编译或测试

Python 修改后：
- 使用 quant-ai/.venv
- 运行 compileall 或相关测试

Vue 修改后：
- 读取 package.json 中实际脚本
- 运行 TypeScript 检查和生产构建

## 每次任务完成后输出

1. 修改摘要
2. 修改文件列表
3. 数据库变化
4. 接口变化
5. 已执行测试
6. 测试结果
7. 未解决问题
8. 下一步建议
