# Stock Quant Pro 最终产品总规划

> 本文定义 Stock Quant Pro 最终应成为怎样的产品，以及达到什么条件才可认定产品真正完成。本文描述目标蓝图，不代表所有能力当前均已实现。

## 1. 文档定位与权威关系

Stock Quant Pro 的项目文档按以下层级共同发挥作用：

- 本文负责定义最终产品蓝图：面向谁、解决什么问题、用户如何使用、最终具备哪些能力，以及什么条件才算完成。
- [`docs/agent-team/CURRENT_STATE.md`](agent-team/CURRENT_STATE.md) 是当前进度、已实现能力和现实边界的唯一事实来源。
- [`docs/agent-team/ROADMAP.md`](agent-team/ROADMAP.md) 定义开发阶段、依赖关系和下一阶段入口。
- [`docs/agent-team/DECISIONS.md`](agent-team/DECISIONS.md) 和 [`docs/06-agent-team-architecture.md`](06-agent-team-architecture.md) 冻结智能体团队的架构原则、权威边界和不可突破的安全约束。
- 各阶段验收文档记录具体阶段的实现范围、测试证据、已知限制和验收结果。

如本文中的最终目标与当前实现存在差异，必须以 `CURRENT_STATE.md` 判断当前能力，不得把规划能力提前宣传为已完成。

本文不是当前状态报告、单阶段开发任务、底层技术清单、自动炒股方案或收益承诺。

## 2. 产品定位与默认方向

Stock Quant Pro 最终定位为：

> 面向个人专业投资研究的、可审计、可回放、证据驱动的股票研究与风险决策辅助平台。

产品将数据治理、全市场扫描、单股研究、策略回测、公告风险、模拟持仓风险和六智能体协同分析连接成完整工作流。系统形成结构化研究结论和风险提示，最终交易决定始终由用户人工完成。

第一正式产品方向冻结如下：

- 第一正式市场范围以沪深主板为主；其他市场必须经过独立的数据、规则和产品验收后再扩展。
- Windows 单机、单用户桌面应用是第一正式交付形态。
- 支持版本化的不同研究周期，不把产品永久锁定在单一周期。
- 候选池采用“系统形成候选建议、用户确认入池”的模式。
- 最终支持由用户主动开启的本地定时扫描，同时保留手工运行。
- 产品止于研究、模拟交易和可导出的人工计划，不连接券商、不控制证券客户端、不自动下单。
- LLM 不是产品完成的必要条件，只是规则与审计体系成熟后的可选解释层。

### 2.1 核心产品原则

1. 事实先于观点。
2. 数据质量先于评分。
3. 证据先于结论。
4. 风险约束先于积极判断。
5. 可回放先于“看起来聪明”。
6. 确定性规则先于语言模型解释。
7. 缺数时安全降级，不补造事实。
8. 历史结果不可覆盖，只能由新数据版本或新规则版本产生新结果。
9. 研究、模拟交易和真实交易严格分离。
10. 单一收益率不能作为产品成功与否的唯一标准。

## 3. 目标用户与使用场景

### 3.1 目标用户

主要用户是希望建立系统化研究流程的个人投资者，包括：

- 需要管理本地行情数据质量的研究者；
- 希望从全市场中形成可解释候选池的用户；
- 重视历史验证、风险控制和复盘纪律的交易者；
- 希望使用多个专业规则协同分析，但不把最终权力交给 AI 的用户；
- 需要长期保留判断依据、版本和结果的个人研究者。

### 3.2 核心使用场景

- 每日更新证券、行情、日历、公告和公司行动等研究数据；
- 检查数据缺失、异常、来源与版本；
- 对目标市场执行正式全市场扫描；
- 确认系统候选建议并建立候选池；
- 对候选股票开展单股研究和智能体分析；
- 检查公告、事件、模拟持仓和组合风险；
- 执行策略回测、版本比较和历史重放；
- 在影子运行中观察系统判断而不产生交易动作；
- 按周、月、季度复盘规则稳定性、数据质量和决策过程。

## 4. 用户每天的完整使用流程

### 4.1 启动与健康检查

用户启动桌面应用后，首页首先展示 Java、Python、数据库和前端状态，数据库版本、数据新鲜度、最近完整交易日、数据任务异常、最新扫描、研究任务和风险事件摘要。

任一核心服务异常时，系统必须明确指出不可用范围，不得用演示数据冒充真实研究数据。

### 4.2 数据更新与质量门禁

用户在数据管理中心检查证券池、交易日历、行情、公告和公司行动的数据版本，发起允许的数据更新任务，并查看来源、覆盖范围、采集时间、失败项和重试情况。

扫描或研究前，系统检查证券记录、交易日期、行情数量、精确日期命中、OHLC与成交量合法性、来源一致性、复权口径、未来数据和历史时点保证。数据质量被阻断时，下游专业结论不得继续计算。

### 4.3 全市场扫描与候选池

用户选择市场范围、交易日期、扫描规则版本、数据版本和正式运行模式。系统展示扫描证券数量、成功、过滤、失败、重试数量及结构化过滤原因。

系统可以形成候选建议，但股票必须经用户确认后才能进入候选池。每个候选保留来源扫描任务、入池规则、原始排名、数据版本、用户标签、研究状态和移除原因。候选池不是买入清单。

扫描既可手工运行，也可由用户主动开启本地定时运行。定时运行必须可暂停、可审计，并清楚区分正式、测试和重试任务。

### 4.4 单股研究与智能体分析

用户在单股研究中心查看证券属性、历史行情、技术指标、市场级事实、扫描来源、公告事件、策略回测、模拟持仓和历史任务，然后选择交易日期、规则版本和执行模式创建团队任务。

Java 冻结上下文、计算 `contextHash`、预分配六个 run；Python 在一次团队调用中执行固定规则；Java校验后原子持久化任务、run、evidence、veto和decision；前端展示六个专业结果及总控汇总。

### 4.5 模拟持仓与风险

系统检查模拟账户中的仓位、现金、集中度、T+1、止损、止盈、流动性、停牌、公告事件和数据过期风险。风险事件由用户确认和处理。

用户可以建立模拟计划、人工确认模拟成交，或导出供人工核对的研究计划；系统不连接券商，也不修改真实账户。

### 4.6 长期复盘

用户按周期复盘数据稳定性、扫描覆盖率、候选池变化、finding分布、正式否决、模拟交易纪律、最大回撤、规则版本差异、历史回放一致性及人工覆盖原因。复盘目标是改进数据、规则和研究纪律，而不是只挑选盈利案例。

## 6. 最终系统架构

```text
Vue / Electron Windows桌面应用
              │
              ▼
Java Spring Boot权威服务
  ├─ 用户请求、任务身份与状态机
  ├─ 数据治理、扫描、回测与模拟持仓
  ├─ 上下文冻结、contextHash与Python调用
  ├─ 响应校验、事务与权威持久化
  └─ 历史查询、回放、导出与展示API
              │
              ├──────────► PostgreSQL
              │             ├─ 数据集、证券状态与交易日历版本
              │             ├─ 行情、公告与公司行动
              │             ├─ 扫描、回测、候选与模拟持仓
              │             └─ 智能体任务、证据、否决与决策
              │
              ▼
Python无状态智能体团队
  ├─ 接收Java冻结的只读contextSnapshot
  ├─ 执行固定版本的确定性规则
  ├─ 返回六个专业run及其证据引用
  └─ 返回独立总控汇总
```

### 6.1 权威职责

Java 始终负责任务与run身份、状态转换、上下文冻结、`contextHash`、请求与响应校验、数据库事务、Evidence/Veto/Decision权威持久化和历史重放。

Python 始终无状态，不直接访问智能体业务数据库，不修改任务身份和冻结上下文，不掌握持久化权，不执行交易操作，不依赖外部网络、随机数或隐式运行状态形成权威结论。

PostgreSQL 保存不可覆盖的事实和版本、point-in-time所需的有效时间与知识时间，以及任务、上下文、证据、否决、报告、扫描、回测和模拟持仓结果。

Vue/Electron 负责展示、用户输入、任务控制、人工确认、比较和复盘，前端不得直接调用Python智能体接口。

## 7. 六个智能体与总控的最终职责

| 角色 | 最终职责 | 主要输入 | 主要输出 | 正式否决权 |
|---|---|---|---|---|
| DATA_QUALITY | 判断数据是否完整、合法、版本明确且无前视 | 全部上下文的数据质量事实 | gate、finding、evidence、异常原因 | 否 |
| MARKET_REGIME | 对市场级事实进行确定性环境分类 | 市场行情、历史证券池、市场宽度、交易日历 | 市场状态、score、confidence、evidence | 否 |
| TECHNICAL_ANALYSIS | 分析目标证券价格、成交量和技术结构 | 个股行情与技术指标 | 趋势、动量、波动和关键位置finding | 否 |
| STRATEGY_BACKTEST | 解释与任务匹配的冻结回测 | 可验证回测上下文 | 样本、稳定性、回撤和适用限制 | 否 |
| ANNOUNCEMENT_RISK | 识别公告、公司行动和事件风险 | 结构化公告与事件 | 风险事件、时间窗口和证据 | 否 |
| POSITION_RISK | 评估模拟持仓、集中度、流动性和风险约束 | 持仓、现金与风险规则 | 仓位风险、限制条件、正式否决 | 是 |
| 总控 | 汇总六个专业run形成团队报告 | 六个run及其evidence、gate和veto | finalDecision、汇总finding和来源run | 不单独新增否决 |

总控不是第七个专业run。DATA_QUALITY的 `BLOCKED` 是数据门禁，不是正式风险否决；正式 `veto=true` 永远只属于POSITION_RISK。任何积极结论都不得绕过正式否决。

## 8. 从数据采集到最终报告的完整链路

1. 数据源产生证券、日历、行情、公告或公司行动事实。
2. 数据管理模块记录来源、版本、采集时间、业务生效时间、系统知晓时间和哈希。
3. Java将合法数据写入不可覆盖的历史事实层或版本化投影。
4. 正式扫描基于冻结证券池、交易日期和规则版本运行。
5. 系统形成候选建议，用户确认后建立候选池。
6. 用户创建单股智能体研究任务。
7. Java在只读一致性事务内冻结九类上下文：`security`、`marketData`、`marketBreadth`、`scanResult`、`technicalMetrics`、`backtestContext`、`securityEvents`、`portfolioContext`和`dataQualityContext`。
8. Java规范化上下文并计算 `contextHash`，为六个专业智能体预分配run身份。
9. Python按团队规则版本执行规则；DATA_QUALITY先形成门禁。
10. 各智能体只引用合法、白名单化的Evidence；POSITION_RISK可按规则形成正式Veto。
11. 总控汇总六个run，但不得超越未实现或不可用的专业能力。
12. Java依据冻结契约和确定性校验规则，独立验证身份、状态组合、数值范围、证据引用、正式否决与最终决策一致性；需要双实现校验的核心规则由对应阶段单独冻结。
13. 校验通过后，任务、run、evidence、veto和decision在事务中原子持久化。
14. Vue展示任务状态、六个run、证据链、限制、否决和总控报告。
15. 用户人工决定继续研究、加入观察、建立模拟计划、导出人工计划或放弃。
16. 后续复盘可定位到原始数据版本、上下文哈希和规则版本。

#### 数据中心

展示证券、行情、交易日历、公告、公司行动的数据规模、来源、数据集版本、有效时间、知识时间、可信等级、异常和修订记录。用户可执行更新、校验和安全重试。

#### 全市场扫描与候选池

#### 单股研究

展示证券身份、状态历史、K线、成交量、技术指标、市场宽度、扫描来源、公告事件、回测摘要、模拟持仓和历史任务。所有图表显示截止日期、复权口径、来源和版本。

#### 策略回测

展示策略版本、参数、数据版本、样本范围、交易明细、收益、回撤、胜率、换手、样本量和适用限制。允许版本并列比较，不允许自动寻优后直接发布生产规则。

#### 智能体团队工作台

页面包括任务摘要、冻结上下文摘要、数据质量门禁、六智能体结果、Evidence浏览器、正式Veto区、总控报告以及版本与重放信息。

每个智能体卡片展示agentCode、status、gateStatus、decision、score、confidence、veto、summary、findings、evidence引用、errors和ruleVersion。

总控区域明确展示sourceRunIds、DATA_QUALITY门禁、POSITION_RISK正式否决、不可用专业能力和报告适用范围。

#### 公告、模拟持仓与风险

公告页面展示结构化事件时间线、来源、发布时间、系统知晓时间、类型、影响和证据。模拟持仓页面展示模拟资金、计划、待确认委托、模拟成交、持仓、可卖数量、风险事件和权益曲线。任何模拟成交仍需人工确认。

#### 历史任务、影子运行与评测复盘

历史任务支持按股票、日期、规则版本、数据版本和状态检索、比较、导出和重放。影子运行只记录稳定性、漂移、失败、缺数和风险表现，不产生订单。评测复盘展示固定评测集、版本对比、无前视违规、重放一致性、finding质量、veto质量和人工复核标签。

## 10. 数据、证据与审计体系

### 10.1 冻结上下文与contextHash

每个正式任务保存完整、不可变的 `contextSnapshot`。后续数据库变化不得静默改变历史任务含义。

`contextHash`证明上下文内容身份：对象字段规范排序，数值按数学值规范化，数组保持业务顺序，不经过浮点转换破坏精度，PostgreSQL JSONB往返保持一致，且哈希规范化不修改实际快照。

### 10.2 数据和规则版本

每次正式结论至少可追溯到数据集、来源、连接器、证券池、交易日历、行情、公司行动、上下文生产、团队规则、算法和契约版本。规则变更产生新版本，不覆盖旧结果。

### 10.3 Point-in-time与无前视

历史研究必须区分业务事实何时生效、系统何时知道、数据何时修订和请求时点允许看到哪个版本。不能证明当时可知的数据，必须标记为不可用于无前视结论。

### 10.4 Evidence

每条关键finding必须引用Evidence。Evidence字段、必填性、枚举和白名单以冻结Schema为准；当前契约没有冻结为必填的字段不得在本文擅自提升为必填。未来新增字段必须单独升级契约并保持旧任务兼容。

Evidence必须包含足以确认其确定性身份、类别、来源、业务对象、观察时点和内容完整性的冻结字段。Evidence不得包含最终decision、gateStatus、score、confidence、投资建议或未经审计的自然语言推断。

### 10.5 历史不可覆盖与可重复回放

历史任务、run、evidence、veto和decision不可由新规则覆盖。新数据或规则只能创建新任务。在数据版本、上下文版本、规则版本和输入完全一致时，系统应重复产生相同的contextHash、finding、evidenceId、score、confidence、veto和finalDecision。

### 10.6 缺数安全降级

缺数、来源不明、历史时点不可证明或上下文不一致时，系统返回结构化不可用原因和安全状态，不生成方向性finding，不以低分代替未知，也不伪造中性结果。

## 11. 规则引擎与未来LLM的关系

确定性规则负责事实判断、指标、finding、score、confidence、gateStatus、veto、finalDecision、Evidence字段和风险约束。

LLM不是产品完成条件。未来如引入，最多作为默认可关闭、明确标注的解释层，用于解释已有结构化报告和提出后续研究问题。

LLM不能生成事实或行情、补造Evidence、修改评分与门禁、形成正式否决、改变finalDecision或产生交易指令，也不参与权威事务与持久化判断。

## 12. 产品级演进阶段

### 12.1 可靠基础设施

形成稳定的本地运行、任务状态、事务、日志、健康检查、错误处理和安全停止能力。

### 12.2 完整数据治理

完成证券、交易日历、行情、公司行动、公告和版本治理，支持point-in-time查询、历史证券池和无前视回放。

### 12.3 六智能体专业能力

逐一完成六个专业智能体的真实、确定性、版本化规则及其Evidence和Java校验。

### 12.4 总控综合报告

只有六个专业智能体均具备真实能力后，总控才形成完整团队报告，同时保持正式风险否决边界。

### 12.5 影子运行

在不产生交易动作的前提下持续观察数据、规则、失败模式和结论漂移。

### 12.6 评测和长期培养

建立固定评测集、版本比较、人工复核、历史重放和长期复盘。规则升级必须通过固定评测，不得根据少量盈利案例调参。

### 12.7 可选LLM解释层

只有规则、证据、审计和评测体系成熟后，才可增加可关闭的解释层；缺少LLM不影响产品完成认定。

## 14. 最终完成标准

Stock Quant Pro 只有同时满足以下产品级条件，才可认定达到最终规划目标。

### 14.1 数据完成标准

- 目标市场证券池具有版本化历史；
- 上市、退市、ST、active和board变化可按时点查询；
- SSE/SZSE交易日历可按知识截止时间回放；
- 行情、公司行动和复权处理不存在未声明的前视；
- 来源、版本、采集时间、生效时间和修订历史可追溯；
- 缺失、停牌、未上市、退市和数据故障可以区分；
- 正式任务不使用演示数据或未知来源数据。

### 14.2 分析完成标准

- 六个专业智能体均具备真实、确定性、版本化规则；
- 每个finding均有合法Evidence；
- DATA_QUALITY可以阻断不可靠分析；
- POSITION_RISK是唯一正式veto来源；
- Java能够独立验证Python权威输出的冻结契约；
- 总控不会超越专业智能体实际能力；
- 不同规则版本可以并列比较且不覆盖旧结果。

### 14.3 回放和审计完成标准

- 任一正式任务可定位到完整数据和规则版本；
- 相同版本重复执行得到相同业务结果；
- JSONB持久化前后语义和contextHash一致；
- 历史任务不可覆盖，数据修订产生新版本和新结果；
- 失败任务无部分Evidence、Veto或Decision残留；
- 用户可从最终结论逐级追溯到Evidence和冻结事实。

### 14.4 产品流程完成标准

- 用户可以完成从数据更新到长期复盘的闭环；
- 每个异步任务都有状态、进度、错误和重试入口；
- 页面明确区分正式、测试、重试、模拟和影子运行；
- 用户能够管理候选池和研究状态；
- 用户能够创建、比较、导出和重放智能体任务；
- 本地定时扫描必须由用户主动开启并可随时停用；
- 模拟委托与成交必须人工确认；
- 人工计划可导出，但系统不连接券商；
- 风险事件可追踪、处理和复盘。

### 14.5 质量和交付完成标准

- 关键规则具有边界、异常、契约和跨语言测试；
- 真实PostgreSQL、Java和Python生产链路完成闭环验收；
- 数据与规则升级具有固定回归门禁；
- 评测集覆盖缺数、停牌、公司行动、历史修订和未来数据注入；
- 系统能够恢复失败任务而不破坏历史；
- Windows单机桌面安装、升级、数据目录、备份、恢复、日志、服务生命周期和崩溃恢复经过验收。

### 14.6 成功评价标准

产品成功不能只用收益率衡量，还必须评价数据完整性、无前视违规、结果可重复性、Evidence完整性、风险识别质量、正式否决质量、规则漂移、系统可用性、任务失败原子性、报告可理解性和用户人工决策纪律。

## 15. 永久能力边界

无论产品演进到哪个阶段，以下边界永久有效：

1. 不承诺收益，不宣传稳定盈利或自动赚钱。
2. 不编造行情、指标、公告、新闻、持仓或Evidence。
3. 不允许LLM成为事实、评分、风险或决策权威。
4. 不允许LLM产生正式Veto或交易指令。
5. 不自动下单，不连接或直接控制券商账户和证券客户端。
6. 不直接修改真实账户现金、持仓和成交记录。
7. 所有模拟成交必须由用户人工确认。
8. 最终交易决定由用户在系统外独立核对并完成。
9. 不因积极结论绕过DATA_QUALITY门禁。
10. 不因积极结论绕过POSITION_RISK正式否决。
11. 不把缺数映射为中性、看空或任意低分结论。
12. 不把测试、演示、重试或回填推断数据冒充正式事实。
13. 不将单只股票指标冒充市场级事实。
14. 不将回测收益解释为未来收益承诺。
15. 不允许自动寻优规则未经固定评测和人工批准直接进入正式版本。
16. 任何能力无法证明时，系统必须明确降级，而不是扩大结论。
